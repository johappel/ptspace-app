1. Neu versendete Nachrichten werden scheinbar nicht geschickt.
2. Das UI wirkt wie ein Dashboard, nicht wie ein Denkraum.
3. Der Kernel funktioniert, aber Daten werden in der App nicht oder nur teilweise gerendert.
4. Feedback zu laufenden Vorbereitungen (Service Requests) ist nicht einsehbar.
5. Tests prüfen nicht, ob Inhalte wirklich gerendert werden.
6. `docker build` schlägt fehl.

## Gefundene Ursachen und Fixes

### A. Docker-Build schlug fehl (behoben, verifiziert)

**Ursache:** Das Frontend nutzte `@sveltejs/adapter-auto`. Ohne erkennbare
Zielplattform erzeugt dieser Adapter **kein** `frontend/build/`-Verzeichnis. Das
`frontend/Dockerfile` kopierte aber `COPY --from=build /app/frontend/build …` und
brach mit `"/app/frontend/build": not found` ab. Zusätzlich startete der Container
über `pnpm preview` (Vorschaumodus statt Produktionsserver).

**Fix:**
- `@sveltejs/adapter-node` als Frontend-Adapter (`frontend/svelte.config.js`,
  Dev-Dependency ergänzt).
- `frontend/Dockerfile`: Startbefehl `node build/index.js`, Laufzeit-Env
  `HOST=0.0.0.0`, `PORT=5173`; überflüssiges `.svelte-kit`-Copy entfernt.
- `docker-compose.yml`: `HOST`/`PORT` für den Frontend-Service gesetzt.

**Verifikation:** `docker build -f frontend/Dockerfile .` erzeugt jetzt ein
vollständiges Image; der Build meldet „Using @sveltejs/adapter-node ✓ done".

### B. Uneinheitliche Backend-Port-Variable (behoben, verifiziert)

**Ursache:** `backend/src/config.ts` las ausschließlich `process.env.PORT`,
während `docker-compose.yml` und `.env.example` `BACKEND_PORT` setzen. In Docker
wurde `BACKEND_PORT` ignoriert (funktionierte nur zufällig, weil Default und
Port-Mapping beide 5174 sind). Setzt jemand `BACKEND_PORT` abweichend, während
`PUBLIC_BACKEND_URL`/Vite-Proxy auf einen anderen Port zeigen, entsteht genau das
Symptom „Nachrichten kommen nicht an" (Frontend spricht einen toten Port an).

**Fix:** `config.ts` liest jetzt `BACKEND_PORT ?? PORT ?? 5174`. Beide Namen
funktionieren; bestehende lokale `.env` (mit `PORT=3000`) bleibt gültig.

### C. Nachrichten-Sendepfad (Regressionstests ergänzt)

Im Code selbst wurde **kein** Fehler im Sende-/SSE-Parser gefunden; der
Transport-Regressionstest im Backend und der Mock-Harness liefern korrekt. Die
wahrscheinlichsten realen Ursachen für „Nachricht wird nicht geschickt" sind:

- Port-/Proxy-Fehlkonfiguration (siehe B).
- Der reale Harness ist aktiviert, aber nicht bereit (`checkAvailability` ≠
  `ready`): Das Backend sendet dann ein `error`-Event, der Sendeaufruf wirft, die
  optimistische Lehrkraftnachricht bleibt ohne Antwort stehen. Das liest sich wie
  „nicht gesendet".

**Ergänzt:** `frontend/test/apiStream.test.ts` deckt den SSE-Sendepfad ab
(Antwort-Zustellung, gepufferte Einzel-Chunks wie hinter dem Vite-Proxy,
Fehler-Event, Nicht-Stream-Fehlerantwort). So werden künftige Regressionen im
Sendepfad automatisch erkannt.

**Diagnose-Checkliste bei erneutem Auftreten:**
1. Läuft das Backend auf dem Port aus `PUBLIC_BACKEND_URL`? (`/health` prüfen.)
2. Zeigt `/health` `harnessAvailability.status = ready`?
3. Bei realem Harness: Modell/Provider/Timeout in `.env` korrekt?

### D. Fehlende Frontend-Tests für das Rendern (behoben)

**Ursache:** Das Frontend hatte gar keine Tests (`"test": "echo \"No frontend
tests yet\""`). Backend-Tests können nicht erfassen, ob API-Daten im DOM landen.

**Fix:**
- Vitest + Testing-Library + jsdom als Frontend-Testumgebung
  (`frontend/vitest.config.ts`, `frontend/test/setup.ts`).
- Render-Helfer aus der Riesenkomponente in ein testbares Modul ausgelagert
  (`frontend/src/lib/markdown.ts`); `+page.svelte` nutzt es weiter.
- `frontend/test/markdown.test.ts`: prüft, dass Gesprächs-/Denkstand-Inhalte
  wirklich als DOM-Knoten erscheinen und Nutzertext escaped bleibt.
- `frontend/package.json`: `test` → `vitest run`.

**Verifikation:** `pnpm --filter @ptspace/frontend test` → **10/10 grün**.

## G. Mock-Harness statt echtem Betrieb (Ursache für „dumme Mockantwort")

**Symptom:** Jede Nachricht wird mit demselben Platzhaltertext beantwortet; beim
Reload steht eine „Mockantwort".

**Ursache:** Die lokale `.env` wählte den simulierten Harness:

```
PTSPACE_HARNESS=mock
PTSPACE_REAL_HARNESS_ENABLED=false
```

Damit läuft der `MockHarnessAdapter`, der unabhängig vom Input eine feste Antwort
liefert. **Alle automatischen Tests laufen ebenfalls gegen diesen Mock** – sie
prüfen die Verkabelung (Persistenz, SSE, Board, Marker), aber keine echte
Modellkonversation. Das ist die berechtigte Kritik „bildet die Praxisnutzung
nicht ab".

**Zweiter, versteckter Trap:** Selbst mit aktiviertem Harness hätte
`PTSPACE_OPENCODE_MODEL=opencode/deepseek-v4-flash-free` den Provider auf
`opencode` gezwungen. Auf diesem Pfad mountet das Backend den
`OPENROUTER_API_KEY` **nicht** in den Container – der echte Aufruf scheitert an
der Authentifizierung. Der Schlüssel wird nur für OpenRouter-Modelle genutzt
(Provider-Logik in `backend/src/config.ts`: Modell, das **nicht** mit
`opencode/` beginnt → Provider `openrouter`).

**Verifizierte, funktionierende Konfiguration** (Laufzeit auf diesem Rechner
vorhanden: opencode-CLI, Docker-Image `ptspace/opencode-test:1.17.13`, Kernel):

```
PTSPACE_HARNESS=opencode-docker
PTSPACE_REAL_HARNESS_ENABLED=true
PTSPACE_OPENCODE_RUNNER=docker
PTSPACE_OPENCODE_MODEL=tencent/hy3:free      # OpenRouter-Modell, nicht opencode/*
PTSPACE_OPENCODE_ALLOW_NETWORK=true
PTSPACE_EXTERNAL_KERNEL_CONTEXT_ENABLED=true
OPENROUTER_API_KEY=...                        # bereits gesetzt
```

`/health` meldet mit dieser Konfiguration `harnessAvailability.status = "ready"`
(nur lokale Prüfung: `docker --version` + Kernel-Dateien, **kein** Modellaufruf,
keine API-Kosten). Diese Werte wurden in `.env` gesetzt.

**Wichtig:**
- Das Backend liest die Konfiguration **beim Start**. Nach der `.env`-Änderung
  das Backend neu starten (`pnpm dev` stoppen und neu starten).
- Ein echter Gesprächs-Turn verbraucht OpenRouter-Credits und dauert je nach
  Modell ~30–120 s. Während dieser Zeit zeigt die Oberfläche „Ich denke kurz
  mit …".
- `tencent/hy3:free` ist der dokumentierte Default. Für ein anderes Modell einen
  gültigen **OpenRouter**-Modell-Identifier verwenden (z. B.
  `deepseek/deepseek-chat`), nicht `opencode/*`.
- `availability = ready` garantiert nur, dass die Runtime erreichbar ist. Ob ein
  konkreter Modell-Identifier existiert, zeigt sich erst beim ersten echten Turn.

**UI-Schutz gegen Verwechslung:** Läuft der Mock, zeigt die Gesprächsansicht
jetzt einen Hinweis „Simulierter Modus" (`simulatedMode` aus `/health`), damit
ein Platzhalter nie mehr als echte Antwort missverstanden wird.

## Sicherheit

Die lokale `.env` enthält einen **echten `OPENROUTER_API_KEY`** im Klartext. Die
Datei ist zwar gegitignored (nicht im Repo), der Schlüssel wurde aber im Klartext
abgelegt. **Empfehlung: Schlüssel rotieren** und nur in einem Secret-Store bzw.
lokal (nie in Chat, Workspace oder Git) halten (vgl. `AGENTS.md` Regel 4).
